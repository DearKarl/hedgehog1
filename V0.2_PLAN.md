# Hedgehog One v0.2 Plan: CodeGraph Foundation

Hedgehog One / 刺猬一号 v0.2 extends the v0.1a deterministic Diagram IR -> SVG compiler into a deterministic CodeGraph -> Diagram IR -> SVG pipeline.

v0.2 planning status: draft plan, no source implementation.

## 1. Product Goal

v0.2 upgrades Hedgehog One from a Diagram IR compiler into a code relationship visualization pipeline:

```text
CodeGraph IR v0.2a -> validate -> adapt to Diagram IR v0.1a -> v0.1a SVG compiler -> SVG
```

The product goal is to make code structure verifiable before it becomes a diagram. Code relationships must enter a structured CodeGraph IR first, then pass through deterministic validation and mapping into Diagram IR. Final SVG output continues to be generated only by the existing v0.1a compiler path.

v0.2 should start with CodeGraph instead of PPT, HTML, GUI, or multi-agent work because CodeGraph strengthens the core truth layer of the project. A GUI or deck exporter would only make the output easier to view; CodeGraph makes the input more meaningful, testable, and auditable. Multi-agent planning can wait until the deterministic substrate is stronger.

## 2. Scope

v0.2 must include:

- CodeGraph IR v0.2a schema

- CodeGraph -> Diagram IR adapter

- tests and fixtures
- documentation

Optional later-half scope:

- import dependency graph only
- file and line evidence on every extracted edge

## 4. CodeGraph IR v0.2a Draft

Root fields:

- `codeGraphVersion`
- `id`
- `title`
- `language`
- `symbols`
- `edges`
- `metadata`

Symbol fields:

- `id`
- `label`
- `kind`: `module | file | function | class | method | value`
- `file` optional
- `line` optional

Edge fields:

- `id`
- `from`
- `to`
- `kind`: `imports | calls | contains | depends_on`
- `label` optional
- `evidence` optional

Evidence fields:

- `file`
- `line`
- `column` optional
- `text` optional

Initial schema expectations:

- unknown fields are rejected
- required fields are explicit
- IDs use the same SVG-safe pattern as Diagram IR where possible
- `edge.from` and `edge.to` are references checked by semantic validation
- evidence is preserved exactly and not resolved through the file system or network

## 5. CodeGraph -> Diagram IR Mapping

The adapter maps CodeGraph IR v0.2a into Diagram IR v0.1a without generating SVG directly.

Symbol to node role mapping:

- `module` -> `source`
- `file` -> `source`
- `function` -> `transform`
- `class` -> `model`
- `method` -> `transform`
- `value` -> `metric`

Edge mapping:

- `imports` -> Diagram IR edge with label `imports`
- `calls` -> Diagram IR edge with label `calls`
- `contains` -> Diagram IR edge with label `contains`
- `depends_on` -> Diagram IR edge with label `depends_on`

Evidence mapping:

- each CodeGraph edge with evidence receives a stable `evidenceRef`
- `evidenceRef` is generated deterministically from the edge id, for example `codegraph:<edge-id>`
- `metadata.evidence[evidenceRef]` stores the evidence object
- evidence is not parsed, fetched, enriched, or silently repaired

ID and ordering rules:

- CodeGraph ids must be SVG-safe or rejected
- generated Diagram IR ids must be SVG-safe
- symbol array order determines node order
- edge array order determines edge order
- adapter output uses stable object field order
- mapping must not use randomness, current time, network, file system, or unordered object enumeration

## 6. CLI Draft

Planned commands, not implemented in this planning checkpoint:

```sh
hedgehog1 codegraph validate input.codegraph.json
hedgehog1 codegraph to-diagram input.codegraph.json -o output.diagram.json
hedgehog1 codegraph compile input.codegraph.json -o output.svg
```

Optional extractor command:

```sh
hedgehog1 code imports src -o output.svg
```

Command semantics:

- `codegraph validate` parses and validates CodeGraph IR only
- `codegraph to-diagram` validates CodeGraph IR and writes deterministic Diagram IR v0.1a
- `codegraph compile` validates CodeGraph IR, adapts to Diagram IR, and reuses the v0.1a SVG compiler
- `code imports` may later extract TypeScript / JavaScript import edges before entering CodeGraph IR

## 7. Testing Strategy

v0.2 tests must cover:

- schema tests
- semantic validation tests
- adapter mapping tests
- deterministic output tests
- invalid fixtures
- evidence preservation tests
- CLI tests in the later CLI milestone

Minimum fixture set:

- valid module import graph
- valid file dependency graph
- duplicate symbol id
- duplicate edge id
- unknown edge source
- unknown edge target
- invalid symbol kind
- invalid edge kind
- invalid id
- missing required field
- evidence preservation fixture

Determinism acceptance:

- the same CodeGraph input produces byte-identical Diagram IR
- the same CodeGraph input produces byte-identical SVG through the v0.1a compiler
- equivalent object field order produces identical output when array order is unchanged
- invalid input produces diagnostics and does not write output

## 8. Milestones

### M0 v0.2 plan + docs

- Create v0.2 planning docs.
- Do not create source code.
- Do not modify v0.1a compiler logic.

### M1 CodeGraph IR schema

- Add CodeGraph IR v0.2a TypeBox schema.
- Add published JSON schema.
- Add schema fixtures and schema-level tests.

### M2 CodeGraph validation

- Add parse and validation entry points for CodeGraph IR.
- Add stable diagnostics and error codes.
- Validate duplicate IDs, missing references, unsupported kinds, and invalid evidence shape.

### M3 CodeGraph -> Diagram IR adapter

- Implement deterministic adapter from validated CodeGraph IR to Diagram IR v0.1a.
- Preserve symbol and edge order.
- Map evidence into `evidenceRef` and `metadata.evidence`.

### M4 codegraph CLI

- Add planned `hedgehog1 codegraph ...` commands.
- Keep CLI as orchestration only.
- Reuse v0.1a compiler for final SVG output.

### M5 optional TypeScript import extractor

- Add TypeScript / JavaScript import dependency extractor if still desired.
- Extract import graph only.
- Preserve file and line evidence.
- Do not implement call graph or CFG.

### M6 v0.2 deterministic acceptance

- Add end-to-end CodeGraph determinism tests.
- Add real dist CLI smoke tests.
- Confirm no LLM and no network runtime path.

## 9. Principles

- An LLM must never directly generate code flow SVG.
- Code relationships must first enter structured CodeGraph IR.
- CodeGraph IR must be verifiable.
- CodeGraph -> Diagram IR must be deterministic.
- Final SVG remains generated by the v0.1a compiler.
- Invalid input must produce explicit diagnostics.
- Invalid input must not be silently repaired.
- Every feature must have automated tests.
- Runtime must not depend on an LLM or network.
